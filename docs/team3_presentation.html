04 Generic Document Abstraction

One pipeline. Swappable schema per doc type.

class OperationalDocument:
  # PO · TO · WO · SHIPMENT · INVOICE
  doc_type: DocType
  references: List[EntityRef]
    # PO number, supplier ID, SKU…
  line_items: List[LineItem]
    # qty, delivery_date, status, price
  status_updates: List[StatusUpdate]
  metadata: {
    "source_email", "confidence",
    "tenant_id", "timestamp"
  }

Plugin pattern per doc_type

  • Each type registers: extraction prompt, field validators, ERP write handler
  • Adding a new doc type = config + prompt, not new code
PO Update Transfer Order Work Order Shipment Notice Invoice + next type…

06 Safeguards

Reliability = known accuracy + traceable failures

The scariest failure mode: a confident but wrong write silently corrupts a production PO. Defense-in-depth.

Audit Trail
Every ERP write carries: source email hash, extraction trace, confidence score, operator ID.
Soft Writes
Updates staged as "pending" for a configurable window (e.g. 15 min) before commit.
Anomaly Detection
Flag writes deviating from historical patterns — quantity 10× normal, delivery date 2 years out.
Rollback
Pending writes reversed instantly. Committed writes log a compensating transaction.
Confidence Gates
Per-tenant thresholds, not global. Aerospace ≠ retail risk tolerance.
The Key Insight
An ops team can tolerate 2% wrong answers they can fix in 2 minutes. They cannot tolerate 0.1% wrong answers discovered at month-end inventory count.
Reliability isn't just accuracy. It's known accuracy with traceable failures. An ops team can tolerate 2% wrong answers if they can find and fix them in 2 minutes. They cannot tolerate 0.1% wrong answers that they discover in a month-end inventory count.

09 Design Decisions

Key tradeoffs — be ready for Q&A

DecisionChoiceWhy
Shared vs per-tenant LLMShared base + per-tenant few-shot cacheCost; fine-tune only after sufficient data
Sync vs async ERP writesAsync (staged) by defaultRollback window + avoids ERP rate limits
Rule-based vs LLM extractionLLM for unstructured, rules for structured fieldsRegex on PO numbers is cheaper & more reliable
Human-in-loop thresholdConfigurable per tenant, not globalAerospace ≠ retail risk tolerance
Schema generalizationAbstract doc model + plugin per typeAvoid fork-per-customer maintenance hell
"What if an LLM hallucinates a PO number?"
Matching engine won't find a record → confidence drops → escalated to human, not auto-applied.
"Supplier changes their email format?"
Correction rate spikes → drift alert → human reviews a few → new few-shot examples → model adapts.
"What about GDPR / data residency?"
Tenant-isolated data store + regional deployment; email content purged post-extraction.
"How long until learning adds real value?"
50–100 confirmed examples per tenant — most active supply chains hit this in 1–2 weeks.
These are the most common pushbacks. Be ready for them. The hallucination one is important — the architecture handles it gracefully by design.
